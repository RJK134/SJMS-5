### Architecture

```

  scripts/
    generate-synthetic-dataset.mjs   ← reads schema, generates CSVs
    sjms-data/
      generators/                    ← one file per topical area
```

Run pipeline:

```sh
# 1. Generate locally (deterministic — same seed = same bytes)
SJMS_DATASET_SEED='2026-05' \
node scripts/generate-synthetic-dataset.mjs --out output/2026-05-17

# 2. Sync to the lake
rclone sync output/2026-05-17/ gdrive5tb:sjms-5-dataset/2026-05-17/
rclone sync output/2026-05-17/ gdrive5tb:sjms-5-dataset/latest/

# 3. SJMS-5 imports (Phase 0+ — once SJMS-5 has a Prisma DB)
cd RJK134/SJMS-5 && pnpm import-sjms-dataset
```

### Generator design principles

1. **Deterministic.** Seed the RNG once at the top; pass it down. Same seed = byte-identical CSVs across reruns. Lets reviewers diff dataset versions cleanly.
2. **Topological order.** Generate parents before children (Course → CourseRun → ModuleRun → Enrolment → Assessment). Wrong order causes FK references to row ids that don't exist yet.
3. **Realistic UK demographics.** Use real distributions: ONS age cohorts, IMD deciles, school-type splits, HESA disability/ethnicity bands, mature-learner shares. Don't invent — there's open data for everything.
4. **Internally consistent.** A student row implies an application row that resolved to an offer that became an enrolment. Don't generate isolated rows. The generator's outer loop should be `for each student → fan out the supporting rows`.
5. **Use the FORBIDDEN_COLUMNS rule from Maieus.** Even though SJMS-5 has full learner data legally, the lake CSV must NOT contain copyrighted past-paper text, mark-scheme text, etc. Same rule, same enforcement.
6. **Manifest required.** `manifest.json` records `{ generatedAt, seed, generatorCommit, rowCounts: { table: n }, schemaVersion }`. The importer rejects mismatched schemas loudly.

### Recommended phase ordering for the SJMS-5 dataset work

1. **Phase D0** — verify the 199-model schema is stable. Read `RJK134/sjms-v4-integrated/server/prisma/schema.prisma`. List every model. Group by domain (Identity, Curriculum, Admissions, Enrolment, Assessment, Finance, Research, HR, Accommodation, Reporting). ~half day.
2. **Phase D1** — write `generate-synthetic-dataset.mjs` skeleton: arg parsing, RNG, manifest, CSV writer, topological-order driver. Generators are empty stubs. Output: a manifest with 199 zero-row tables. ~half day.
3. **Phase D2** — fill Identity + Curriculum (students, applicants, staff, courses, modules). Most volume; gets the FK skeleton right. ~1-2 days.
4. **Phase D3** — Admissions + Enrolment (applications, offers, decisions, enrolments). ~1 day.
5. **Phase D4** — Assessment + Progression + Award (the v4 strength). ~1-2 days.
6. **Phase D5** — Finance (fees, invoices, payments, sponsors, bursaries). ~1-2 days.
7. **Phase D6** — Research + HESA artefacts + everything else. ~1-2 days.
8. **Phase D7** — write SJMS-5's importer (`pnpm import-sjms-dataset`). Mirror Maieus's `import-datalake.mjs` shape: thin CLI calling a function from `@sjms-5/db` that walks the CSVs in topological order and upserts. ~1 day.
9. **Phase D8** — wire BullMQ scheduler for periodic refresh (optional — datasets rarely change in practice, so manual run is fine for v1). ~half day.
