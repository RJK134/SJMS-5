### `.github/workflows/k6-nightly.yml` — new

- Triggers: nightly cron `0 3 * * *` (03:00 UTC, low-traffic window for UK HE staging) + manual `workflow_dispatch` (with `scenario` and `base_url` inputs).
- Soft-skips when `K6_BASE_URL` secret is not configured, with a clear `::warning::` so the operator notices the unset state without the job failing.
- `continue-on-error: true` — the job is advisory through Phase 0, ratcheting to required in Phase 12 (pilot readiness) alongside the 2× peak rehearsal requirement.
- Default scenario for nightly runs: `smoke-test.js` (the cheap one: 1 VU smoke → 10 VU baseline → 50 VU stress, total ~3 minutes).
- Heavier scenarios (`admissions-peak.ts`, etc.) are operator-dispatched only via `workflow_dispatch` to avoid silently ramping 5,000 VUs against staging every night.
- Build step: `node k6/build.cjs` — `esbuild` is already on the root `package.json` from the spine, so no new dependency.
- k6 binary install: `grafana/k6` Debian package via the official `dl.k6.io` apt repo.
- Summary export: `k6 run --summary-export evidence/k6/<scenario>.summary.json` → uploaded as workflow artefact `k6-summary-<run_id>` with 30-day retention.

## Operator action items at merge time

1. **Set the `K6_BASE_URL` repository secret** to the staging API URL (e.g. `https://sjms5-server-staging.vercel.app/api`). Until this is set, the nightly job soft-skips with a `::warning::` line — no result is emitted but no failure either.

2. **Configure a `sjms-5-load-test` confidential client** in the Keycloak `fhe` realm. The current realm JSON ships only the `sjms-client` public + PKCE client which doesn't support the password grant the k6 token helper uses. The load-test client requires `clientAuthenticatorType: client-secret` + a service-account or direct-access-grants enabled. Sequenced to **operator action** rather than a 0E commit so the secret value doesn't leak into the realm JSON.

3. **Set Keycloak load-test credentials** as repo secrets (`K6_KEYCLOAK_CLIENT_SECRET` + optionally `K6_KEYCLOAK_*` overrides) — the workflow currently only injects `BASE_URL`; future iteration will pass the full Keycloak env block once the load-test client lands.

4. **First-run verification**: trigger `workflow_dispatch` manually with `scenario: smoke-test.js` against staging. Confirm `evidence/k6/smoke-test.summary.json` is produced and the workflow summary shows non-zero `total_requests`.

Batch 0E is `done` per the acceptance-signal protocol. The workflow exists; soft-skips cleanly with `K6_BASE_URL` unset; emits a result artefact when configured.

## Out of scope

- Live `sjms-5-load-test` Keycloak client config (operator action — keeps the secret out of the realm JSON).
- Targeting Vercel production (the workflow soft-skips by default; secret is staging-only).
- k6 → Prometheus / Grafana integration (`k6 run --out experimental-prometheus-rw` etc.) — Phase 10 observability batch owns the metrics pipeline.
- Performance thresholds for the heavy scenarios — the imported defaults are sized for a real institutional staging stack; Phase 12 pilot-readiness owns the 2× peak verification.

## Follow-on items

- `k6 run --summary-export` is deprecated since k6 v0.50 in favour of `--summary` with new flag names. The current workflow will likely emit a deprecation warning but still produce the artefact. Phase 12 cleanup should refresh to the new flag.
- Heavier scenarios (api-stress.ts, etc.) can be wired to a separate `k6-peak.yml` workflow with a heavier runner (e.g. self-hosted) in Phase 12.
- The Keycloak token helper in `config.ts` caches per-role tokens but does not refresh on 401 — a Phase 12 hardening pass should add explicit token-refresh on a verification 401.
